Development Task: School Timetable Page (Principal Dashboard) 
Context 
الصفحة المقصودة هي:

لوحظ أن صفحة الجدول المدرسي الحالية لا تحقق الوظيفة المطلوبة بالشكل الصحيح. 
لذلك يجب إعادة مراجعة هذا الجزء من النظام بعناية شديدة قبل تنفيذ أي تعديل. 
1. Code Safety Requirement (مهم جدًا) 
قبل تنفيذ أي تعديل: 
1.	قم بتحليل الكود الحالي المرتبط بصفحة School Timetable بالكامل. 
2.	حدد بوضوح: 
o	ما الذي تم تنفيذه بالفعل. 
o	ما الذي يمكن تعديله. 
o	ما الذي يمكن حذفه لأنه غير مستخدم أو غير ضروري. 
1.	تأكد من عدم حدوث: 
o	تعارض بين الكود القديم والجديد. 
o	ازدواجية في المنطق (Logic Duplication). 
o	كود غير مستخدم (Dead Code). 
1.	إذا كان من الأفضل إعادة بناء الجزء بالكامل بدل تعديله، قم بذلك بطريقة نظيفة وآمنة. 
الهدف هو Code Integrity + Clean Architecture.

المعادلة الأساسية لبناء الجدول 
نفترض: 
•	( C ) = مجموعة الفصول (Classes) 
•	( S ) = مجموعة المواد (Subjects) 
•	( T ) = مجموعة المعلمين (Teachers) 
•	( P ) = مجموعة الفترات الزمنية (Periods / Time Slots) 
•	( D ) = أيام الأسبوع 
نعرّف متغير القرار: 
[ 
X_{c,s,t,p} = 
\begin{cases} 
1 & \text
{إذا تم إسناد المادة } 
s \text
{ للفصل } 
c \text
{ بواسطة المعلم } 
t \text
{ في الفترة } 
p \ 
0 & \text
{غير ذلك} 
\
end{cases}
 
] 
أي أن كل خلية في الجدول هي قرار هل توضع هذه الحصة هنا أم لا. 
المعادلة المفاهيمية التي تحاول الخوارزمية حلها 
الفكرة يمكن تحويلها إلى علاقة رياضية: 
[ 
\
text{Timetable Entry} = 
\text{Class} + \text{Subject} + \text{Teacher} + \text{Valid Time Slot}
 
] 
مع تحقق القيود: 
[ 
\
text{Valid Schedule} = 
(C + S + T + P)
 
] 
بحيث يحقق: 
[ 
\
text{No Conflict} \land 
\text{Teacher Load Constraint} \land 
\text{Subject Frequency Constraint}
 
] 
القيود الرياضية الأساسية 
1️⃣ عدم تعارض المعلم 
لا يمكن أن يدرّس المعلم أكثر من فصل في نفس الوقت. 
[ 
\
sum_{c \in C}\sum_{s \in S} X_{c,s,t,p} \le 1 
\quad \forall t,p
 
] 
2️⃣ عدم تعارض الفصل 
لا يمكن أن يكون للفصل حصتان في نفس الفترة. 
[ 
\
sum_{s \in S}\sum_{t \in T} X_{c,s,t,p} \le 1 
\quad \forall c,p
 
] 
3️⃣ تحقيق عدد الحصص الأسبوعية للمادة 
إذا كانت المادة (s) تحتاج (R_{c,s}) حصص أسبوعياً: 
[ 
\
sum_{t \in T}\sum_{p \in P} X_{c,s,t,p} = R_{c,s}
 
] 
4️⃣ احترام نصاب المعلم (Teacher Load) 
إذا كان الحد الأعلى للحصص الأسبوعية للمعلم هو (L_t): 
[ 
\
sum_{c \in C}\sum_{s \in S}\sum_{p \in P} X_{c,s,t,p} \le L_t
 
] 
5️⃣ توفر المعلم 
إذا كان المعلم غير متاح في فترة معينة: 
[ 
X_{c,s,t,p} = 0
 
] 
6️⃣ توفر الفصل 
إذا كان الفصل غير متاح في فترة معينة: 
[ 
X_{c,s,t,p} = 0
 
] 
دالة الهدف (Objective Function) 
الخوارزمية لا تبحث فقط عن جدول صالح، بل أفضل جدول ممكن. 
مثال لدالة الهدف: 
[ 
\
text{Maximize}
 
] 
[ 
W_1(\text{Teacher Balance}) 
+ 
W_2(\text{Class Balance}) 
+ 
W_3(\text{Preferred Times}) 
W_4(\text{Teacher Gaps}) 
W_5(\text{Subject Clustering})
 
] 
حيث: 
•	(W_i) أوزان تحدد أهمية كل معيار. 
الصيغة المفاهيمية المبسطة التي يمكن استخدامها في النظام 
يمكن التعبير عنها كما يلي: 
[ 
Schedule(c,s,t,p) = 
\begin{cases} 
1 & \text
{إذا تحققت جميع القيود} \ 
0
 & \
text
{غير ذلك} 
\
end{cases}
 
] 
مع: 
[ 
Valid(p,t,c) = 
Availability(p,t) 
\times 
Availability(p,c) 
\times 
LoadConstraint(t) 
\times 
SubjectRequirement(c,s)
 
] 
التفسير العملي للخوارزمية 
بمعنى أبسط: 
الخوارزمية تبحث عن حل يحقق المعادلة: 
[ 
Class + Subject + Teacher + TimeSlot
 
] 
لكن بشرط: 
[ 
\
text{No Conflict}
 
] 
و 
[ 
\
text{Within Teacher Workload}
 
] 
و 
[ 
\
text{Subject Weekly Requirements}
 
] 
الصيغة التنفيذية 
يمكن اختصارها كالتالي: 
TimetableEntry = 
(Class, Subject, Teacher, TimeSlot) 
 
Subject to: 
1) No Teacher Conflict 
2) No Class Conflict 
3) Subject Weekly Requirement 
4) Teacher Workload Limits 
5) Teacher Availability 
6) Class Availability 
7) School Time Blocks

1) الهدف العام من الصفحة 
صفحة الجدول المدرسي داخل حساب مدير المدرسة هي الصفحة المركزية المسؤولة عن: 
•	توليد الجدول المدرسي بالذكاء الاصطناعي 
•	عرض الجدول الناتج بطريقة بصرية واضحة 
•	مراجعة جودة التوزيع 
•	اكتشاف المشكلات والتعارضات 
•	نشر نسخة الجدول أو إبقائها كمسودة 
•	إعادة توليد الجدول بالكامل أو جزئيًا 
•	تصفية الجدول حسب الفصل أو المعلم أو الصف أو اليوم 
•	إعطاء مدير المدرسة سيطرة كاملة على دورة حياة الجدول 
هذه الصفحة يجب أن تكون من أقوى الصفحات بصريًا ووظيفيًا لأن الجدول يعتبر من أهم مكونات النظام. 
2) فلسفة التصميم العامة للصفحة 
يجب أن تعتمد الصفحة على المبادئ التالية: 
2.1 وضوح بصري قوي 
•	الصفحة يجب أن تكون سهلة القراءة 
•	الجدول يجب أن يظهر كعنصر رئيسي وواضح 
•	الأولوية البصرية للجدول نفسه ثم الإجراءات ثم التحليلات 
2.2 منع الإرباك 
•	لا توجد أزرار كثيرة بلا معنى 
•	لا يوجد زر “+ جديد” 
•	الإجراء الرئيسي الوحيد في أعلى الصفحة هو: 
معالجة الجدول بالذكاء الاصطناعي 
2.3 Dynamic State Awareness 
الصفحة يجب أن تتغير حسب حالة الجدول: 
•	لا يوجد جدول بعد 
•	يوجد Draft 
•	يوجد Published 
•	يوجد توليد جارٍ 
•	توجد أخطاء جاهزية 
•	توجد مشاكل في النسخة الحالية 
2.4 Database-Driven UI 
كل ما يظهر في الصفحة يجب أن يكون مبنيًا على بيانات قاعدة البيانات: 
•	إعدادات المدرسة 
•	الأيام 
•	الفترات 
•	الصفوف 
•	الفصول 
•	المواد 
•	المعلمين 
•	نسخ الجدول 
•	نتائج التوليد 
•	diagnostics

5) Generation Status Banner 
هذا القسم يظهر تحت الـ Header مباشرة، ويتغير حسب حالة النظام. 
5.1 الحالات الممكنة 
الحالة A: لا يوجد جدول بعد 
يعرض Banner توضيحي مثل: 
•	لا توجد نسخة جدول مولدة حاليًا 
•	يمكنك بدء التوليد باستخدام الذكاء الاصطناعي بعد التأكد من اكتمال الإعدادات

الحالة E: فشل التوليد

Journey 2: التوليد فشل بسبب نقص البيانات

الخطوات

2.	ينزل إلى Issues & Warnings 
3.	يراجع عدد الحصص غير الموزعة 
4.	يفتح diagnostics 
5.	يقرر: 
o	النشر 
o	أو إعادة التوليد

Build the Principal Timetable Page as a smart, database-driven control center for timetable generation, review, diagnostics, and publishing.

•	a strong page header with school context

•	a primary CTA: “معالجة الجدول بالذكاء الاصطناعي”

•	a powerful timetable grid with interactive cells 
•	insights and quality summary panel 
•	version management section 
•	issues and warnings section 
•	clear modals for generation, partial regeneration, publishing, archiving, diagnostics, and session details 
•	full support for draft vs published lifecycle 
•	clear user journeys from empty state to published timetable 
The experience must be visually strong, clean, intelligent, and operationally realistic.

Every UI responsibility must be isolated in a dedicated component. 
The page container should orchestrate data fetching, state management, and backend actions, while child components remain focused on display logic, user interaction, and event emission. 
The component system must support: 
•	clear data flow 
•	reusable controls 
•	draft/published timetable lifecycle 
•	interactive grid cells 
•	readiness validation 
•	diagnostics visibility 
•	full and partial generation flows

1) API Design Principles 
قبل التفاصيل، يجب الالتزام بهذه القواعد: 
1.1 Context Required in Every Request 
كل API خاص بالجدول يجب أن يعمل ضمن سياق واضح: 
•	school_id 
•	academic_year_id 
•	term_id 
إما: 
•	في path params 
•	أو في query params 
•	أو يتم استنتاجها من session/current school context 
الأفضل: 
تمريرها في query أو body عند الحاجة بوضوح. 
1.2 Response Envelope Standard 
يفضل توحيد شكل الرد: 
{ 
  "success": true, 
  "message": "Request completed successfully",

الهدف من المحرك 
بناء محرك ذكي لتوليد الجدول المدرسي داخل منصة نَسَق | NASSAQ بحيث يقوم بإنشاء جدول دراسي صالح ومنطقي ومرن، اعتمادًا بالكامل على بيانات قاعدة البيانات، مع احترام القيود الأكاديمية والتشغيلية، وتقليل التعارضات، وتحسين جودة التوزيع. 
المحرك لا يجب أن يكون مجرد أداة تعبئة خانات، بل يجب أن يعمل كـ: 
•	Constraint-Aware Engine 
•	AI-Assisted Scheduler 
•	Dynamic Allocation System 
•	Versioned Timetable Generator 
1) الفكرة الأساسية للمحرك 
المحرك يحاول حل المعادلة التالية لكل حصة مطلوبة: 
فصل + مادة + معلم + وقت مناسب + بدون تعارض + ضمن النصاب + ضمن إعدادات المدرسة 
أي أن كل حصة يتم إنشاؤها فقط إذا تحقق الآتي: 
•	الفصل متاح 
•	المادة مطلوبة لهذا الفصل 
•	يوجد معلم صالح لتدريسها 
•	الفترة الزمنية مناسبة ومسموح بها 
•	لا يوجد تعارض مع حصص أخرى 
•	نصاب المعلم لم يتجاوز الحدود 
•	لا يتم خرق قواعد اليوم الدراسي 
•	يتم احترام قواعد الجودة والتوزيع 
2) مصادر البيانات التي يعتمد عليها المحرك

•	إعدادات المدرسة 
•	العام الدراسي 
•	الفصل الدراسي 
•	الأيام الدراسية النشطة 
•	الفترات الزمنية 
•	الحصص التعليمية فقط 
•	الاستراحات 
•	الصلاة 
•	الصفوف والفصول 
•	المواد 
•	عدد الحصص الأسبوعية لكل مادة 
•	ربط المواد بالصفوف 
•	ربط المعلمين بالمواد 
•	قيود توفر المعلمين 
•	قيود توفر الفصول 
•	الحدود القصوى للنصاب 
•	التفضيلات الزمنية 
•	النسخة الحالية من الجدول إن وجدت 
لا يجوز استخدام أي قيم ثابتة داخل الواجهة أو الكود بشكل غير مرتبط بقاعدة البيانات. 
3) مراحل تشغيل المحرك 
المرحلة الأولى: Readiness Check 
قبل أي عملية توليد، يجب تنفيذ فحص جاهزية كامل. 
يتأكد المحرك من: 
•	وجود مدرسة نشطة 
•	وجود عام دراسي نشط 
•	وجود فصل دراسي نشط 
•	وجود أيام أسبوع مفعلة 
•	وجود فترات زمنية صالحة 
•	وجود فترات تعليمية كافية 
•	وجود فصول دراسية 
•	وجود مواد مرتبطة بالصفوف 
•	وجود عدد حصص أسبوعي لكل مادة 
•	وجود معلم واحد على الأقل لكل مادة مطلوبة 
•	وجود إعدادات نصاب واضحة 
•	عدم وجود تناقضات تمنع التوليد 
إذا فشل الفحص: 
•	لا يبدأ التوليد 
•	يعيد قائمة تفصيلية بالأخطاء 
•	يربط كل خطأ بالكيان المسبب له 
•	يعرض للمستخدم ما الذي يجب إصلاحه أولًا 
المرحلة الثانية: Build Scheduling Context 
بعد اجتياز الفحص، يقوم المحرك ببناء سياق التوليد. 
يشمل ذلك: 
•	الأيام التعليمية الفعلية 
•	الفترات التعليمية فقط 
•	الفترات غير التعليمية المستثناة 
•	جميع الفصول المستهدفة 
•	جميع المواد المطلوبة لكل فصل 
•	جميع المعلمين المؤهلين لكل مادة 
•	جميع القيود الصلبة والمرنة 
•	جميع التفضيلات 
الناتج من هذه المرحلة هو نموذج داخلي موحد يستخدمه المحرك أثناء الإسناد. 
المرحلة الثالثة: Build Demand Units 
يقوم المحرك بتحويل الاحتياج الأكاديمي إلى وحدات جدولة مستقلة. 
مثال: 
إذا كان فصل 5A يحتاج: 
•	رياضيات: 5 حصص أسبوعيًا 
•	علوم: 3 حصص أسبوعيًا 
•	عربي: 4 حصص أسبوعيًا 
فالمحرك ينشئ: 
•	5 وحدات جدولة للرياضيات 
•	3 وحدات جدولة للعلوم 
•	4 وحدات جدولة للعربي 
كل وحدة تحتوي على: 
•	class_id 
•	grade_id 
•	subject_id 
•	candidate_teacher_ids 
•	weekly_index 
•	priority_weight 
•	constraints 
•	preferences 
هذه الوحدات هي العناصر الفعلية التي يتم توزيعها داخل الجدول. 
المرحلة الرابعة: Constraint Modeling 
يجب أن يفصل المحرك بين نوعين من القيود: 
أولًا: Hard Constraints 
قيود لا يمكن كسرها إطلاقًا: 
1.	المعلم لا يمكن أن يدرّس حصتين في نفس الوقت 
2.	الفصل لا يمكن أن يحضر حصتين في نفس الوقت 
3.	لا يجوز وضع حصة داخل الاستراحة أو الصلاة 
4.	لا يجوز إسناد مادة لمعّلم غير مربوط بها 
5.	لا يجوز تجاوز الحد الأقصى للحصص اليومية للمعلم 
6.	لا يجوز تجاوز الحد الأقصى للحصص الأسبوعية للمعلم 
7.	لا يجوز وضع الحصة في وقت غير متاح للمعلم 
8.	لا يجوز وضع الحصة في وقت غير متاح للفصل 
9.	يجب تحقيق عدد الحصص الأسبوعية المطلوبة لكل مادة 
10.	لا يجوز وضع حصة خارج حدود اليوم الدراسي 
11.	لا يجوز توليد خلية زمنية معطلة أو غير نشطة 
12.	لا يجوز تعيين غرفة غير مناسبة إذا كانت إدارة الغرف مفعلة 
ثانيًا: Soft Constraints 
يفضل احترامها قدر الإمكان لتحسين جودة الجدول: 
1.	توزيع المادة على أيام متعددة بدل تكديسها 
2.	تقليل الفجوات في جدول المعلم 
3.	تقليل الفجوات في جدول الفصل 
4.	تفضيل المواد الأساسية في أول اليوم 
5.	تجنب المواد الثقيلة في آخر اليوم 
6.	تقليل التكرار المتتالي لنفس المادة 
7.	تحقيق توازن في الحمل اليومي 
8.	احترام أوقات التفضيل الخاصة بالمعلم 
9.	ترك مرونة لبعض الفترات إن كانت المدرسة تريد ذلك 
المرحلة الخامسة: Prioritization Engine 
بعد بناء وحدات الجدولة، يقوم المحرك بترتيبها حسب درجة الصعوبة. 
يجب أن يبدأ بالأصعب أولًا، مثل: 
•	مادة لها معلم واحد فقط 
•	مادة ذات حصص أسبوعية كثيرة 
•	مادة لها قيود زمنية صارمة 
•	فصل مزدحم بعدد كبير من المواد 
•	مادة أساسية 
•	مادة تحتاج غرفة خاصة 
•	معلم نصابه قريب من الحد الأقصى 
الهدف: 
تقليل احتمالية الفشل في المراحل اللاحقة. 
المرحلة السادسة: Candidate Slot Generation 
لكل وحدة جدولة، يقوم المحرك بإنشاء قائمة بالفترات الزمنية المحتملة. 
يتم استبعاد أي فترة إذا: 
•	المعلم مشغول 
•	الفصل مشغول 
•	الوقت غير متاح 
•	الفترة ليست Teaching Slot 
•	المادة ممنوعة في هذا التوقيت 
•	النصاب سيُكسر 
•	عدد مرات المادة في اليوم سيتجاوز الحد 
•	التوقيت خارج القواعد المحددة 
الناتج هو مجموعة فترات مرشحة صالحة مبدئيًا. 
المرحلة السابعة: Slot Scoring Engine 
كل فترة مرشحة يجب أن تحصل على درجة تقييم. 
أمثلة لعناصر التقييم: 
•	المعلم متاح: +30 
•	الفصل متاح: +30 
•	الوقت مفضل للمادة: +15 
•	الوقت مفضل للمعلم: +10 
•	لا يسبب تكرار متتالٍ للمادة: +10 
•	يحقق توازن الحمل اليومي: +10 
•	يقلل الفجوات: +8 
•	لا يضع مادة ثقيلة في آخر اليوم: +7 
•	لا يكدس نفس المادة في نفس اليوم: +10 
ثم تُرتب الفترات من الأعلى إلى الأقل. 
مبدأ مهم: 
المحرك لا يختار أول خانة متاحة فقط، بل يختار أفضل خانة ممكنة. 
المرحلة الثامنة: Iterative Assignment 
يبدأ المحرك في الإسناد الفعلي وحدة بعد أخرى. 
لكل وحدة: 
1.	يأخذ الوحدة ذات الأولوية الأعلى 
2.	يجلب قائمة الفترات المرشحة 
3.	يحسب Score لكل فترة 
4.	يختار أفضل فترة صالحة 
5.	ينشئ timetable_entry 
6.	يحدث حالة الإشغال للمعلم والفصل 
7.	يحدث عدادات النصاب 
8.	يحدث عدادات المادة اليومية والأسبوعية 
ثم ينتقل إلى الوحدة التالية. 
المرحلة التاسعة: Conflict Resolution 
إذا تعذر إسناد وحدة في أي فترة مناسبة، يبدأ المحرك في حلول المعالجة. 
الاستراتيجيات: 
1.	تجربة معلم آخر من المرشحين 
2.	تجربة ثاني أفضل فترة 
3.	Swap مع حصة أقل أولوية 
4.	Backtracking محدود 
5.	إعادة توزيع جزئي 
6.	Relax بعض الـ Soft Constraints فقط 
7.	تسجيل الوحدة كمشكلة unresolved إذا لم يوجد حل 
ممنوع: 
كسر Hard Constraints تحت أي ظرف. 
المرحلة العاشرة: Optimization Pass 
بعد اكتمال التوزيع الأولي، يبدأ المحرك مرحلة تحسين الجودة. 
يفحص: 
•	الأيام المزدحمة 
•	الفجوات 
•	تكرار المادة 
•	توازن نصاب المعلمين 
•	توازن الأحمال بين الأيام 
•	كثافة المواد الأساسية 
•	جودة بداية ونهاية اليوم 
يحاول: 
•	تبديل بعض الحصص 
•	نقل حصة من يوم مزدحم إلى يوم أخف 
•	تحسين تموضع مادة أساسية 
•	تقليل الفراغات

المرحلة الحادية عشرة: Quality Evaluation 
بعد اكتمال التوزيع والتحسين، يحسب المحرك درجة جودة نهائية. 
مثال على مكونات الجودة:

•	صفر تعارضات إلزامية

7) الصيغة التنفيذية المختصرة للمطور 
Timetable AI Engine يجب أن يعمل كالتالي: 
•	Read school context from database 
•	Validate readiness 
•	Build teaching demand units 
•	Build hard and soft constraints 
•	Prioritize units by difficulty 
•	Generate valid candidate slots 
•	Score candidate slots 
•	Assign best slot iteratively 
•	Resolve local conflicts using swap/backtracking 
•	Optimize final distribution 
•	Evaluate timetable quality 
•	Save as draft version 
•	Return diagnostics and summary 
8) Pseudocode تشغيلي 
def generate_timetable(school_id, academic_year_id, term_id, mode="full"): 
    context = load_timetable_context(school_id, academic_year_id, term_id) 
 
    readiness = validate_context(context) 
    if not readiness.is_valid: 
        return build_readiness_error_response(readiness) 
 
    demand_units = build_demand_units(context) 
    demand_units = prioritize_units(demand_units, context) 
 
    version = create_draft_version(context, mode) 
 
    for unit in demand_units: 
        candidate_slots = build_candidate_slots(unit, context, version) 
        scored_slots = score_candidate_slots(unit, candidate_slots, context, version) 
 
        assigned = False 
 
        for slot in scored_slots: 
            if passes_hard_constraints(unit, slot, context, version): 
                assign_unit(unit, slot, version, context) 
                assigned = True 
                break 
 
        if not assigned: 
            assigned = try_fallback_strategies(unit, version, context) 
 
        if not assigned: 
            log_unresolved_issue(version, unit) 
 
    run_optimization_pass(version, context) 
 
    quality = evaluate_timetable_quality(version, context) 
    update_version_quality(version, quality) 
 
    return build_generation_response(version, quality) 
9) الجملة الأساسية التي تحكم المنطق كله 
الخوارزمية تحاول أن تجد لكل حصة مطلوبة: 
أفضل معلم صالح + أفضل وقت صالح + أفضل توزيع ممكن + بدون أي تعارض + مع احترام النصاب والقيود الأكاديمية 
 
 
السياق الأكاديمي مصدر البيانات هي قاعده البيانات من صفحه الاعدادات يجلب لعام الدراسي الحالي والفصل الدراسي الحالي 
أيام الدراسة مصدر البيانات هي قاعده البيانات من صفحه الاعدادات يجلب ايام الدراسة 
هيكل اليوم الدراسي مصدر البيانات هي قاعده البيانات من صفحه الاعدادات يجلب عدد الحصص اليومية و مدة الحصة 
الفصول الدراسية مصدر البيانات هي قاعده البيانات من صفحه الاعدادات يجلب الفصول الدراسية 
المعلمون مصدر البيانات هي قاعده البيانات من صفحه الاعدادات يجلب اسماء المعلمون 
إسنادات المعلمين  مصدر البيانات هي قاعده البيانات من صفحه الاعدادات يجلب إسنادات المعلمين 
 
 
 
 
 
 
Database Schema — School Timetable System 
1) Core Academic Context Tables 
هذه الجداول تحدد السياق العام الذي يعمل داخله الجدول. 
Table: schools 
CREATE TABLE schools ( 
    id BIGINT PRIMARY KEY GENERATED ALWAYS AS IDENTITY, 
    name_ar VARCHAR(255) NOT NULL, 
    name_en VARCHAR(255), 
    code VARCHAR(100) UNIQUE NOT NULL, 
    timezone VARCHAR(100) NOT NULL DEFAULT 'Asia/Riyadh', 
    status VARCHAR(50) NOT NULL DEFAULT 'active', 
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP, 
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP 
); 
Table: academic_years 
CREATE TABLE academic_years ( 
    id BIGINT PRIMARY KEY GENERATED ALWAYS AS IDENTITY, 
    school_id BIGINT NOT NULL REFERENCES schools(id) ON DELETE CASCADE, 
    name_ar VARCHAR(255) NOT NULL, 
    name_en VARCHAR(255), 
    start_date DATE NOT NULL, 
    end_date DATE NOT NULL, 
    is_active BOOLEAN NOT NULL DEFAULT FALSE, 
    status VARCHAR(50) NOT NULL DEFAULT 'active', 
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP, 
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP 
); 
Table: terms 
CREATE TABLE terms ( 
    id BIGINT PRIMARY KEY GENERATED ALWAYS AS IDENTITY, 
    school_id BIGINT NOT NULL REFERENCES schools(id) ON DELETE CASCADE, 
    academic_year_id BIGINT NOT NULL REFERENCES academic_years(id) ON DELETE CASCADE, 
    name_ar VARCHAR(255) NOT NULL, 
    name_en VARCHAR(255), 
    code VARCHAR(50), 
    start_date DATE NOT NULL, 
    end_date DATE NOT NULL, 
    is_active BOOLEAN NOT NULL DEFAULT FALSE, 
    status VARCHAR(50) NOT NULL DEFAULT 'active', 
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP, 
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP 
);

ON timetable_entries(timetable_version_id, subject_id, weekday_number, time_block_id);

12) Suggested Integrity Rules 
يفضل أيضًا إضافة قواعد منطقية على مستوى قاعدة البيانات: 
منع تعارض المعلم داخل نفس النسخة 
CREATE UNIQUE INDEX uq_teacher_one_slot_per_version 
ON timetable_entries (timetable_version_id, teacher_id, weekday_number, time_block_id) 
WHERE teacher_id IS NOT NULL AND entry_status <> 'cancelled'; 
منع تعارض الفصل داخل نفس النسخة 
CREATE UNIQUE INDEX uq_class_one_slot_per_version 
ON timetable_entries (timetable_version_id, class_id, weekday_number, time_block_id) 
WHERE class_id IS NOT NULL AND entry_status <> 'cancelled'; 
منع تكرار نفس البلوك أكثر من مرة للفصل في نفس النسخة 
CREATE UNIQUE INDEX uq_class_block_once_per_version 
ON timetable_entries (timetable_version_id, class_id, weekday_number, time_block_id, entry_type) 
WHERE class_id IS NOT NULL AND entry_status <> 'cancelled';

Algorithm توزيع الحصص 
إليك Algorithm توزيع الحصص بصياغة تنفيذية دقيقة جدًا، مناسبة مباشرة للـ Vibe Coding Agent داخل مشروع نَسَق | NASSAQ. 
Algorithm توزيع الحصص 
School Timetable Distribution Algorithm 
الهدف 
بناء خوارزمية ذكية لتوزيع الحصص داخل الجدول المدرسي بحيث تقوم بتوليد جدول صالح ومنطقي ومتوازن، اعتمادًا على: 
•	الفصول 
•	المواد 
•	المعلمين 
•	الفترات الزمنية 
•	القيود الأكاديمية 
•	النصاب 
•	التفضيلات 
•	جودة التوزيع 
الخوارزمية يجب ألا تكون توزيعًا عشوائيًا، بل يجب أن تعمل كالتالي: 
Constraint-Based Scheduling + Heuristic Prioritization + Iterative Assignment + Local Conflict Resolution + Optimization Pass 
1) الفكرة الأساسية للخوارزمية 
الخوارزمية تحاول حل المعادلة التالية لكل حصة مطلوبة: 
فصل + مادة + معلم صالح + وقت صالح + بدون تعارض + ضمن النصاب + ضمن قيود المدرسة 
أي أن كل حصة لا تُنشأ إلا إذا تحقق الآتي: 
•	المادة مطلوبة لهذا الفصل 
•	يوجد معلم صالح لها 
•	هذا المعلم متاح 
•	هذا الفصل متاح 
•	الفترة الزمنية صالحة 
•	لا يوجد تعارض 
•	لا يتم كسر النصاب 
•	لا يتم كسر القواعد اليومية 
•	يتم اختيار أفضل توزيع ممكن وليس مجرد أول مكان متاح 
2) نوع الخوارزمية 
الخوارزمية المطلوبة هي: 
Constraint-Based Heuristic Scheduler 
وتعمل عبر 5 طبقات: 
1.	Preparation Layer 
تجهيز البيانات والسياق 
2.	Demand Modeling Layer 
تحويل الاحتياجات إلى وحدات قابلة للجدولة 
3.	Constraint Filtering Layer 
استبعاد كل الاختيارات غير الصالحة 
4.	Scoring & Assignment Layer 
تقييم أفضل الفترات وإسنادها 
5.	Repair & Optimization Layer 
إصلاح التعارضات وتحسين جودة الجدول 
3) المدخلات المطلوبة 
الخوارزمية تعمل فقط إذا توفرت البيانات التالية من قاعدة البيانات: 
•	المدرسة 
•	العام الدراسي 
•	الفصل الدراسي 
•	الأيام الدراسية المفعلة 
•	الفترات التعليمية 
•	الفصول 
•	الصفوف 
•	المواد 
•	عدد الحصص الأسبوعية لكل مادة 
•	ربط المواد بالصفوف 
•	ربط المعلمين بالمواد 
•	توفر المعلمين 
•	توفر الفصول 
•	قواعد النصاب 
•	التفضيلات 
•	قيود المادة 
•	قيود الغرف إن كانت مفعلة 
•	إعدادات الجدول الافتراضية 
4) المخرجات المطلوبة 
الخوارزمية يجب أن تنتج: 
•	جدول كامل أو جزئي 
•	قائمة الحصص الموزعة 
•	قائمة الحصص غير الموزعة 
•	عدد التعارضات 
•	درجة جودة التوزيع 
•	تحذيرات تشغيلية 
•	Diagnostics واضحة 
•	نسخة Draft من الجدول 
5) منطق العمل العام 
الخوارزمية تنفذ المراحل التالية بالتسلسل: 
المرحلة 1: Load Active Context 
تحميل كل السياق النشط من قاعدة البيانات: 
•	المدرسة الحالية 
•	العام الدراسي الحالي 
•	الفصل الدراسي الحالي 
•	الأيام الدراسية النشطة 
•	الفترات التعليمية فقط 
•	الفصول النشطة 
•	المواد المطلوبة لكل صف 
•	المعلمين المرتبطين بكل مادة 
•	القيود 
•	التفضيلات 
•	النصاب 
•	النسخة الحالية إذا لزم 
الهدف 
بناء Scheduling Context موحد وواضح. 
المرحلة 2: Validate Readiness 
قبل البدء، يجب التأكد من أن البيانات كافية لتوليد جدول. 
يتم التحقق من: 
•	وجود أيام دراسية 
•	وجود Teaching Slots 
•	وجود فصول 
•	وجود مواد لكل صف 
•	وجود عدد حصص أسبوعية لكل مادة 
•	وجود معلمين صالحين لكل مادة 
•	وجود قيود نصاب 
•	عدم وجود تناقضات تمنع التوليد 
إذا فشل الفحص: 
•	يتم إيقاف الخوارزمية 
•	إرجاع قائمة بالأسباب 
•	عدم إنشاء جدول ناقص بصمت 
المرحلة 3: Build Demand Units 
هذه أهم خطوة. 
بدل أن تتعامل الخوارزمية مع "مادة" بشكل عام، يجب تحويل كل احتياج إلى Scheduling Unit مستقلة. 
مثال 
فصل 5A يحتاج: 
•	رياضيات = 5 حصص 
•	علوم = 3 حصص 
•	عربي = 4 حصص 
يتم إنشاء: 
•	5 وحدات رياضيات 
•	3 وحدات علوم 
•	4 وحدات عربي 
كل Unit يجب أن تحتوي على: 
•	unit_id 
•	class_id 
•	grade_id 
•	subject_id 
•	candidate_teacher_ids 
•	weekly_required_count 
•	weekly_index 
•	priority_weight 
•	constraints 
•	preferences 
مثال بنية الوحدة: 
{ 
  "unit_id": "CLS12-SUB8-03", 
  "class_id": 12, 
  "grade_id": 3, 
  "subject_id": 8, 
  "candidate_teacher_ids": [15, 21], 
  "weekly_required_count": 5, 
  "weekly_index": 3, 
  "priority_weight": 92, 
  "constraints": { 
    "max_per_day": 1, 
    "allow_consecutive": false 
  } 
} 
المرحلة 4: Prioritize Units by Difficulty 
لا يجب توزيع الوحدات عشوائيًا. 
يجب ترتيبها من الأصعب إلى الأسهل. 
معايير الصعوبة: 
•	عدد المعلمين المتاحين أقل 
•	عدد الفترات الممكنة أقل 
•	المادة أساسية 
•	المادة لها حصص أسبوعية كثيرة 
•	المادة لها قيود زمنية صارمة 
•	الفصل مزدحم 
•	المعلم قريب من سقف النصاب 
•	المادة تتطلب غرفة خاصة 
الهدف 
ابدأ بالوحدات الأصعب حتى لا تتبقى فقط الحالات المستحيلة في النهاية. 
المرحلة 5: Build Candidate Slots 
لكل وحدة، يجب إنشاء قائمة الفترات الزمنية المحتملة. 
Candidate Slot = 
(weekday + time_block + teacher + class) 
لكن لا يتم قبول أي slot إلا إذا اجتاز Hard Constraints. 
يتم استبعاد أي Slot إذا: 
•	المعلم مشغول 
•	الفصل مشغول 
•	الوقت غير متاح للمعلم 
•	الوقت غير متاح للفصل 
•	الفترة ليست Teaching Slot 
•	النصاب سيتجاوز الحد 
•	المادة تجاوزت الحد اليومي 
•	المادة ممنوعة في ذلك التوقيت 
•	الغرفة المطلوبة غير متاحة 
•	توجد حصة أخرى لنفس الفصل أو المعلم في نفس الوقت 
المرحلة 6: Score Candidate Slots 
بعد تكوين القائمة الصالحة مبدئيًا، يتم تقييم كل Slot. 
هذه الخطوة هي جوهر "الذكاء" في التوزيع. 
معادلة التقييم العامة: 
Slot Score = 
Availability Score 
+ Distribution Score 
+ Preference Score 
+ Load Balance Score 
+ Gap Reduction Score 
+ Subject Logic Score 
- Penalty Score 
مثال للأوزان: 
•	teacher available = +30 
•	class available = +30 
•	preferred subject time = +15 
•	preferred teacher time = +10 
•	no consecutive repetition = +10 
•	balanced daily load = +10 
•	reduce teacher gap = +8 
•	reduce class gap = +8 
•	avoid heavy subject late = +7 
•	prevent same subject clustering = +10 
•	penalty for overload = -50 
•	penalty for poor placement = -10 
النتيجة 
يتم ترتيب كل الفترات من الأعلى Score إلى الأقل. 
المرحلة 7: Iterative Assignment 
تبدأ الخوارزمية في الإسناد الفعلي وحدة بعد أخرى. 
لكل Unit: 
1.	خذ أعلى وحدة أولوية 
2.	اجلب Candidate Slots 
3.	احسب Score لكل Slot 
4.	اختر أعلى Slot صالح 
5.	أنشئ entry في الجدول 
6.	حدّث إشغال المعلم 
7.	حدّث إشغال الفصل 
8.	حدّث عداد حصص المادة في اليوم 
9.	حدّث عداد نصاب المعلم 
10.	انتقل للوحدة التالية 
مبدأ مهم 
لا يتم فقط اختيار أول خانة متاحة، بل أفضل خانة ممكنة وقت التنفيذ. 
المرحلة 8: Constraint Recheck After Assignment 
بعد كل عملية إسناد، يجب إعادة فحص الحالة العامة. 
يتم تحديث: 
•	teacher occupancy 
•	class occupancy 
•	daily subject counters 
•	teacher daily load 
•	teacher weekly load 
•	class daily density 
•	room occupancy إذا كانت مفعلة 
هذا يمنع أن تبني الخوارزمية قرارات لاحقة على بيانات قديمة. 
المرحلة 9: Fallback Strategies 
إذا لم تتمكن الخوارزمية من إسناد وحدة ما، لا يجب أن تتوقف فورًا. 
يجب أن تستخدم استراتيجيات معالجة بالتدريج. 
الترتيب المقترح: 
1.	جرّب ثاني أفضل Slot 
2.	جرّب معلمًا آخر من المرشحين 
3.	جرّب يومًا آخر 
4.	جرّب فترة أخرى أقل تفضيلًا 
5.	فعّل Swap مع حصة أقل أولوية 
6.	نفذ Backtracking محدود 
7.	رخّص بعض Soft Constraints فقط 
8.	إذا استمر الفشل، سجل unresolved issue 
ممنوع تمامًا: 
كسر أي Hard Constraint. 
المرحلة 10: Swap Logic 
إذا لم يوجد مكان صالح مباشر، يمكن استخدام Swap. 
مثال: 
•	وحدة رياضيات لا تجد مكانًا 
•	توجد حصة نشاط في Slot مناسب 
•	يمكن نقل حصة النشاط إلى Slot آخر دون ضرر 
•	يتم تحرير هذا الـ Slot لوحدة الرياضيات 
شروط الـ Swap: 
•	لا يكسر أي Hard Constraint 
•	لا يسبب سلسلة تعارضات غير قابلة للحل 
•	يفضل أن يكون مع وحدة أقل أولوية 
•	يتم تقييمه قبل اعتماده 
المرحلة 11: Limited Backtracking 
إذا استمر الفشل، تستخدم الخوارزمية Backtracking محدود، وليس شاملًا حتى لا يصبح الأداء بطيئًا. 
المنطق: 
•	ارجع عددًا محدودًا من الخطوات 
•	فك آخر 1 إلى 3 إسنادات مرتبطة بالتعارض 
•	جرّب حلولًا بديلة 
•	أعد البناء الجزئي 
الهدف 
إنقاذ الجدول دون إعادة البناء الكامل. 
المرحلة 12: Optimization Pass 
بعد الإسناد الأولي، تبدأ مرحلة تحسين الجودة. 
يتم فحص: 
•	الفجوات في جداول المعلمين 
•	الفجوات في جداول الفصول 
•	تكدس نفس المادة 
•	عدم توازن الأحمال 
•	سوء توزيع المواد الأساسية 
•	ازدحام أيام معينة 
•	ضعف بداية أو نهاية اليوم 
يتم تنفيذ: 
•	نقل حصة من يوم مزدحم إلى يوم أخف 
•	تبديل حصص لتحسين التوازن 
•	تقليل فراغات المعلم 
•	تقليل تكرار المادة في نفس اليوم 
•	تقديم المواد الأساسية إلى فترات أفضل 
المرحلة 13: Quality Evaluation 
بعد اكتمال التوزيع والتحسين، يجب حساب Quality Score. 
المقاييس: 
•	نسبة اكتمال الحصص المطلوبة 
•	عدد التعارضات الإلزامية 
•	توازن حمل المعلمين 
•	توازن حمل الفصول 
•	توزيع المواد على الأيام 
•	تقليل الفجوات 
•	احترام التفضيلات 
•	جودة تموضع المواد الأساسية 
مثال: 
Quality Score = 
40% Completion 
+ 20% No Hard Conflicts 
+ 10% Teacher Balance 
+ 10% Class Balance 
+ 10% Subject Distribution 
+ 10% Soft Constraints Satisfaction 
المرحلة 14: Save Result as Draft Version 
بعد انتهاء التوليد: 
•	لا يتم النشر مباشرة 
•	يتم إنشاء نسخة Draft 
•	يتم حفظ جميع Entries 
•	يتم حفظ Diagnostics 
•	يتم حفظ المشكلات 
•	يتم حفظ الدرجة النهائية 
•	يتم حفظ Logs كاملة 
النتيجة 
يمكن لمدير المدرسة: 
•	مراجعة الجدول 
•	اعتماد الجدول 
•	إعادة التوليد 
•	أرشفة النسخة 
•	العودة لنسخة سابقة 
6) Hard Constraints 
هذه قيود لا يمكن كسرها إطلاقًا: 
1.	لا يمكن أن يدرّس المعلم حصتين في نفس الوقت 
2.	لا يمكن أن يحضر الفصل حصتين في نفس الوقت 
3.	لا يمكن وضع حصة داخل الاستراحة أو الصلاة 
4.	لا يمكن إسناد مادة لمعّلم غير مربوط بها 
5.	لا يمكن تجاوز النصاب اليومي للمعلم 
6.	لا يمكن تجاوز النصاب الأسبوعي للمعلم 
7.	لا يمكن استخدام وقت غير متاح للمعلم 
8.	لا يمكن استخدام وقت غير متاح للفصل 
9.	لا يمكن وضع مادة أكثر من الحد اليومي المسموح 
10.	لا يمكن تجاوز حدود اليوم الدراسي 
11.	لا يمكن استخدام Slot غير نشط 
12.	لا يمكن تعيين غرفة غير مناسبة إذا كانت الغرف مفعلة 
7) Soft Constraints 
هذه قيود تحسين جودة: 
1.	توزيع المادة على عدة أيام 
2.	تقليل الفجوات 
3.	تقليل التكرار المتتالي 
4.	تفضيل المواد الأساسية في الصباح 
5.	تجنب المواد الثقيلة في آخر اليوم 
6.	موازنة أحمال المعلمين 
7.	موازنة أحمال الفصول 
8.	احترام تفضيلات المعلمين 
9.	تحسين بداية ونهاية اليوم الدراسي 
8) قواعد توزيع ذكية يجب تطبيقها 
Rule 1 
المواد الأساسية مثل: 
•	عربي 
•	إنجليزي 
•	رياضيات 
•	علوم 
يجب أن تحصل على أولوية أعلى في الفترات الأولى. 
Rule 2 
إذا كانت المادة تحتاج 5 حصص أسبوعيًا، لا يجب وضعها كلها في يومين فقط إلا عند الضرورة. 
Rule 3 
لا تجعل نفس المادة تتكرر في فترتين متتاليتين لنفس الفصل إلا إذا كان ذلك مسموحًا صراحة. 
Rule 4 
تجنب أن يكون لدى المعلم فراغات طويلة غير مبررة. 
Rule 5 
تجنب أن يبدأ أو ينهي الفصل يومه دائمًا بمادة ثقيلة. 
Rule 6 
إذا تعذر الحل الكامل، أخرج أفضل حل ممكن مع تقرير واضح بالمشكلات. 
9) أوضاع تشغيل الخوارزمية 
Full Generation 
بناء جدول كامل من الصفر. 
Partial Regeneration 
إعادة توليد: 
•	فصل واحد 
•	معلم واحد 
•	يوم واحد 
•	مادة واحدة 
Assist Mode

def generate_timetable(school_id, academic_year_id, term_id, mode="full"): 
    context = load_context(school_id, academic_year_id, term_id) 
 
    readiness_result = validate_readiness(context) 
    if not readiness_result.is_valid: 
        return { 
            "status": "failed", 
            "reason": "readiness_validation_failed", 
            "issues": readiness_result.issues 
        } 
 
    demand_units = build_demand_units(context) 
    prioritized_units = prioritize_units_by_difficulty(demand_units, context) 
 
    version = create_draft_timetable_version(context, mode) 
 
    occupancy_state = initialize_occupancy_state(context, version) 
 
    for unit in prioritized_units: 
        candidate_slots = build_candidate_slots(unit, context, occupancy_state) 
        scored_slots = score_candidate_slots(unit, candidate_slots, context, occupancy_state) 
 
        assigned = False 
 
        for slot in scored_slots: 
            if passes_hard_constraints(unit, slot, context, occupancy_state): 
                assign_unit(unit, slot, version, occupancy_state) 
                assigned = True 
                break 
 
        if not assigned: 
            assigned = try_fallback_strategies(unit, version, context, occupancy_state) 
 
        if not assigned: 
            log_unresolved_issue(version, unit, "unassigned_unit") 
 
    run_optimization_pass(version, context, occupancy_state) 
 
    quality = evaluate_quality(version, context, occupancy_state) 
    save_quality_metrics(version, quality) 
 
    return { 
        "status": "success", 
        "version_id": version.id, 
        "quality_score": quality.total_score, 
        "assigned_units": quality.assigned_units, 
        "unassigned_units": quality.unassigned_units, 
        "warnings": quality.warnings 
    }

def assign_unit(unit, slot, version, occupancy_state): 
    create_timetable_entry( 
        version_id=version.id, 
        class_id=unit.class_id, 
        grade_id=unit.grade_id, 
        subject_id=unit.subject_id, 
        teacher_id=slot.teacher_id, 
        weekday_number=slot.weekday_number, 
        time_block_id=slot.time_block_id, 
        entry_type="lecture", 
        is_ai_generated=True 
    ) 
 
    occupancy_state.mark_teacher_busy( 
        teacher_id=slot.teacher_id, 
        weekday_number=slot.weekday_number, 
        time_block_id=slot.time_block_id 
    ) 
 
    occupancy_state.mark_class_busy( 
        class_id=unit.class_id, 
        weekday_number=slot.weekday_number, 
        time_block_id=slot.time_block_id 
    ) 
 
    occupancy_state.increment_teacher_daily_load( 
        teacher_id=slot.teacher_id, 
        weekday_number=slot.weekday_number 
    ) 
 
    occupancy_state.increment_teacher_weekly_load( 
        teacher_id=slot.teacher_id 
    ) 
 
    occupancy_state.increment_subject_daily_count( 
        class_id=unit.class_id, 
        subject_id=unit.subject_id, 
        weekday_number=slot.weekday_number 
    ) 
12) Pseudocode لوظيفة التقييم 
def score_candidate_slot(unit, slot, context, occupancy_state): 
    score = 0 
 
    if slot.teacher_available: 
        score += 30 
 
    if slot.class_available: 
        score += 30 
 
    if is_preferred_time_for_subject(unit.subject_id, slot, context): 
        score += 15 
 
    if is_preferred_time_for_teacher(slot.teacher_id, slot, context): 
        score += 10 
 
    if not causes_consecutive_same_subject(unit, slot, occupancy_state): 
        score += 10 
 
    if improves_daily_balance(unit, slot, occupancy_state): 
        score += 10 
 
    if reduces_teacher_gaps(slot.teacher_id, slot, occupancy_state): 
        score += 8 
 
    if reduces_class_gaps(unit.class_id, slot, occupancy_state): 
        score += 8 
 
    if avoids_late_heavy_subject(unit.subject_id, slot, context): 
        score += 7 
 
    if causes_subject_clustering(unit, slot, occupancy_state): 
        score -= 10 
 
    return score

14) الصياغة المختصرة الجاهزة للـ Agent 
Build a constraint-based timetable distribution algorithm that: 
•	reads all active school scheduling data from the database 
•	builds weekly demand units for each class-subject requirement 
•	sorts units by difficulty 
•	generates valid candidate slots 
•	scores candidate slots using heuristics 
•	assigns the best valid slot iteratively 
•	uses fallback strategies such as swap and limited backtracking 
•	runs a post-assignment optimization pass 
•	computes a final quality score 
•	saves the result as a draft timetable version with diagnostics

3) Orchestrator Pseudocode 
def generate_timetable(school_id, academic_year_id, term_id, generation_mode, initiated_by): 
    context = load_timetable_context(school_id, academic_year_id, term_id) 
 
    readiness = validate_timetable_readiness(context) 
    if not readiness["is_valid"]: 
        return { 
            "status": "failed", 
            "reason": "readiness_validation_failed", 
            "issues": readiness["issues"] 
        } 
 
    run = create_generation_run(context, initiated_by, generation_mode) 
    version = create_draft_timetable_version(context, initiated_by, generation_mode) 
 
    try: 
        scheduling_model = build_scheduling_model(context) 
        units = build_demand_units(context, scheduling_model) 
        units = prioritize_demand_units(units, scheduling_model) 
 
        occupancy_state = initialize_occupancy_state(context, version["id"]) 
 
        for unit in units: 
            candidate_slots = build_candidate_slots(unit, scheduling_model, occupancy_state) 
            scored_slots = score_candidate_slots(unit, candidate_slots, scheduling_model, occupancy_state) 
 
            assigned = assign_unit_to_best_slot( 
                unit, scored_slots, version["id"], occupancy_state, scheduling_model 
            ) 
 
            if not assigned: 
                assigned = resolve_unassigned_unit( 
                    unit, version["id"], scheduling_model, occupancy_state 
                ) 
 
            if not assigned: 
                log_generation_issue( 
                    run["id"], 
                    "unassigned_unit", 
                    "high", 
                    "scheduling_unit", 
                    None, 
                    "Could not assign this unit to any valid slot", 
                    unit 
                ) 
 
        run_post_generation_optimization(version["id"], scheduling_model, occupancy_state) 
 
        quality = evaluate_timetable_quality(version["id"], scheduling_model, occupancy_state) 
        update_timetable_version_metrics(version["id"], quality) 
        mark_generation_run_completed(run["id"], quality) 
 
        return { 
            "status": "success", 
            "version_id": version["id"], 
            "quality_score": quality["total_quality_score"], 
            "summary": quality 
        } 
 
    except Exception as e: 
        mark_generation_run_failed(run["id"], {"error": str(e)}) 
        raise

5) الصياغة المختصرة    
Build the timetable backend as modular services. 
The engine should include: 
•	context loading 
•	readiness validation 
•	draft version creation 
•	generation run tracking 
•	scheduling model construction 
•	demand unit generation 
•	difficulty-based prioritization 
•	candidate slot generation 
•	hard-constraint validation 
•	slot scoring 
•	iterative assignment 
•	fallback resolution 
•	post-generation optimization 
•	quality evaluation 
•	diagnostics and audit logging 
•	orchestration through a main generation service 
Each responsibility must be isolated in a dedicated backend function or service for maintainability, debuggability, and future scaling.
